## 10/22 项目立项

### 技术栈选择

在立项之初，我便确定了“前端 Vue 后端 Python”的技术架构。这么选择主要是因为：

- 我对 Python 比较熟悉，之前曾使用 `fastapi` 为个人项目搭建过后端。
- 我对 Vue 略有耳闻，我希望学习使用 Vue 。

总的来说，是出于自身已掌握技术与兴趣选取的技术栈。

## 10/23 引入 NuxtUI 组件库

~~我从来没觉得写前端开心过~~

坦率地说，手搓样式对前端还远远没有达到入门水准的我来说是一件很困难的事情。为了在有限的时间内完成项目，我决定放弃手搓组件，转而引入组件库。

根据 [Vue 官网推荐的组件库列表](https://ui-libs.vercel.app/)，我选择了 [NuxtUI](https://ui.nuxt.com/) 。

引入 NuxtUI 对开发效率的提升是立竿见影的：

- 其提供了大量的样式美观、开箱可用的组件
- 无需再自行实现 Vue 相关底层逻辑编写（比如 v-model 的双向绑定）

## 10/24-10/26 调整前端设计方案

得益于 NuxtUI ，我在编写界面上节省了很多时间与工作量。这使得我不满足于仅仅构建一张表单，我开始思考在这之外的拓展性。
为了实现新的构想，我设计了以下的路由结构：

- `/` 首页 显示 Tab
- `/form/{id}` 表单页 显示表单
- `error/{code}` 错误页 显示错误(主要是 404) **_TODO_**

页面设计已初步完成，接下来的一段时间内我会继续完善相关内容，然后开始编写后端。

# 10/27 编写后端

已初步完成后端编写工作，并在前端界面中接入后端。由于这只是一个 demo ，后端保持了相对较低的复杂性。~~能跑就行~~

在演示阶段，我没有使用任何 SQL 数据库，而是使用 json 搭建了一个 **“伪数据库”** 的数据结构。使用 SQL 是可行的，但对目前的我来说这需要引入 SQLModel 这个 ORM 库。（手写 SQL 语句对目前的我来说仍旧有一定难度）。权衡之后，我决定放弃使用 SQL ，而是使用 json 来进行基本演示。

此外，我省略了登陆鉴权等步骤，这主要是因为再搓一个登陆界面略微有些复杂而且有点偏离主题了。

接下来我将尝试编写 Dockerfile 。

# 10/28-10.29 编写 Dockerfile

其实之前有使用过 docker ，但当时只部署了 fastapi 一个后端，文件相对简单。这个项目我希望在一个容器中同时运行前后端，因此借助了一些 AI 的帮助。

我确实没有手动在 Linux 上手动安装过 Nodejs ，所以最终安装 Nodejs 的命令完全是由 AI 生成的。

此外， `start.sh` 也是在 AI 的帮助下完成的，在此之前我的确不知道 bash 中还有 `wait` 这种命令。
具体运行方法请参见 [自述文件](./README.md#使用-docker)
